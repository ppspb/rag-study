# Исследование датасета как топологии retrieval-задачи

## 1. Почему мы начинаем не с модели, а со структуры

До первого retrieval run полезнее понять не «какую модель взять», а
**какую форму имеет сама задача**. Для IR-датасета это значит:

- какой у нас корпус документов;
- какие типы запросов;
- как устроены `qrels`;
- насколько плотная и перекошенная у нас supervision topology;
- какие поля документа реально несут retrieval signal.

Идея простая: если topology коллекции непонятна, то почти любой baseline
будет объясняться магией, а не причиной.

## 2. Декларативный взгляд

Вместо раннего framework-shopping фиксируем декларации:

- **dataset id** — что именно исследуем;
- **field contract** — какие текстовые поля считаем индексируемыми;
- **query contract** — какой текст считаем запросом;
- **relevance contract** — какие метки считаем положительными;
- **profile outputs** — какие артефакты обязательно сохраняем.

Это позволяет обсуждать retrieval-поведение через явные контракты,
а не через «мне кажется этот датасет лёгкий / сложный».

## 3. Топологическая карта датасета

Смотри на датасет как на граф:

- `Docs` — множество документов;
- `Queries` — множество запросов;
- `Qrels` — ребра релевантности `query -> doc`;
- `Fields` — внутренние узлы сигнала (`title`, `text`, `body`, `abstract`);
- `Labels` — веса на рёбрах.

Практические вопросы к этой карте:

1. Насколько разрежены qrels?
2. Сколько релевантных документов в среднем на запрос?
3. Длинные ли документы?
4. Насколько короткие и keyword-like запросы?
5. Есть ли большая неоднородность по датасетам?

## 4. Что измерять до индексации

Минимум:

- `docs_count`
- `queries_count`
- `qrels_count`
- `qrels_per_query`
- длины документов
- длины запросов
- гистограмма relevance labels
- 3-5 живых примеров документов и запросов

Это и есть первый слой Evaluation-Driven Development:
мы сначала измеряем структуру задачи, потом выбираем baseline.

## 5. Типовые failure modes

- маленький корпус, но длинные документы -> BM25 может вести себя иначе,
  чем на веб-поиске;
- короткие keyword-запросы -> dense path может не показать раннего преимущества;
- мало qrels на запрос -> метрики становятся нестабильнее;
- разные датасеты в одном бандле -> средний score скрывает разные режимы отказа.

## 6. Что должно появиться в git после первого прохода

- `configs/datasets.yaml`
- `outputs/dataset_profile_*.json`
- `outputs/dataset_profile_*.csv`
- краткая заметка: что в topology выглядит рискованным для baseline

Только после этого есть смысл переходить к первой индексации.
